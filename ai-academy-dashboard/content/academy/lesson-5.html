2 Governance → Architecture Mapping

🔑 Key Insight

Your governance document is not just paperwork. It is your architecture specification. Every governance decision eliminates or requires certain design patterns.

1. Role Model → Agent Topology

How you define roles in governance directly determines your system's structure.

Governance Decision Architecture Consequence
ROLE "Human must approve every action" → Human-in-the-loop pattern required
ROLE "Validation agents check before execution" → Multi-agent with validator layer
ROLE "One agent handles everything" → Single agent with tool use
ROLE "Agents observe each other" → Supervisor/worker pattern

2. Trust Boundaries → Data Access Patterns

Where your agent can read/write is determined by governance constraints.

Governance Decision Architecture Consequence
TRUST "Agent cannot access customer DB directly" → RAG pattern (read-only vector copy)
TRUST "Agent can create ServiceNow tickets" → Tool Use pattern with API integration
TRUST "Agent must use only approved data sources" → Grounding with allow-list
TRUST "Full authority in sandbox only" → Separate environments, promotion flow
💡

This is where Day 3 connects. Your Supabase vector database — that's not just a learning exercise. It's an architectural answer to the trust boundary problem. If the agent cannot touch production data, you give it a vector copy instead.

3. Failure Protocol → Fallback Architecture

Your error handling architecture is literally defined by your failure protocol.

Governance Decision Architecture Consequence
FAIL "If LLM unavailable → show cached response" → Cache layer required
FAIL "If confidence < 70% → escalate to human" → Confidence scoring + routing
FAIL "If API fails → retry 3x then degrade" → Circuit breaker pattern

Write the governance first, the code writes itself.

3 Four Agent Design Patterns

These are the four main patterns you'll use in 90% of Kyndryl AI projects. They are not theoretical — these are what FDEs deploy at client sites.

📚

Note: More patterns exist (Reflection, Multi-Agent Debate, Chain-of-Thought, Tree-of-Thought, Autonomous Agents). These 4 are the enterprise foundation — you'll encounter others as you specialize.

🧠

ReAct

Reason + Act

▼
Think step-by-step, then act, then observe
ReAct Cycle
🧠
THINK
→
⚡
ACT
→
👁
OBSERVE
↺
ReAct Example
THINK   → "I need to find the user's order status"
ACT     → call_database(order_id=12345)
OBSERVE → result: {status: "shipped", tracking: "CZ123456"}
THINK   → "I have the info, I can respond"
ACT     → respond_to_user("Your order was shipped...")
When to Use
Tasks requiring reasoning before action Multi-step problems Agent must "figure out" what to do
🔒 Governance Connection
  • Behavior Rules define WHAT the agent can reason about
  • Trust Boundaries define WHICH actions are allowed
  • Audit Mode logs each think-act-observe cycle
📚

Real Example: A customer asks: "What's the cheapest way to migrate our Oracle DB to Azure?" The agent needs to reason: What's the DB size? SLA? Budget? ReAct lets it ask follow-up questions, search documentation, compare options, and recommend.

Real Example: The Praveen demo from Day 3 — his ARX agent uses Tool Use pattern. It takes raw notes, calls Power Automate to process them, calls Word Online to generate a document, calls OneDrive to save it.

🚀

Planning / Orchestration

Multi-agent coordination

▼
Break tasks into subtasks, coordinate agents
Planning Example
User: "Prepare a security assessment for our new chatbot"

PLAN:
  Step 1: [AI-SEC agent] → Run OWASP checklist against the code
  Step 2: [AI-DA agent]  → Analyze usage logs for anomalies
  Step 3: [FDE agent]    → Test prompt injection vulnerabilities
  Step 4: [AI-PM agent]  → Compile findings into client report
  Step 5: [Human]        → Review and approve

EXECUTE: Steps 1-3 in parallel, Step 4 after all complete, Step 5 always last
When to Use
Complex multi-skill tasks Cross-functional work Team project (Weeks 4-5) End-to-end automation
🔒 Governance Connection
  • Role Model from Day 4 is literally the orchestration blueprint
  • "Who does what" in governance = "which agent handles which subtask"
  • Human-in-the-loop checkpoints = planning breakpoints
💡

This is the pattern behind KAF's AI Innovation Lab Method: Discovery → Co-Design → Build → Verify → Release → Roadmap to Operate. The governance document you created yesterday is the specification for this orchestration.

Key Discovery Questions for Clients

Ask these questions to understand customer needs before choosing a pattern:

📄 Knowledge & Data
  • What information does the agent need to answer questions?
  • Is this information in documents, databases, or APIs?
  • How often does this information change?
  • Is there proprietary data that cannot be in the base LLM?
Points to: RAG
🔧 Actions & Integrations
  • What systems does the agent need to interact with?
  • Does it need to CREATE, UPDATE, or DELETE anything?
  • What APIs/tools are available?
  • What happens if an external system is unavailable?
Points to: Tool Use
🧠 Reasoning & Decisions
  • Does the agent need to make decisions based on multiple factors?
  • Are there multi-step problems where each step depends on the previous?
  • Does the agent need to "figure out" what to do (not just follow a script)?
  • How complex is the decision logic?
Points to: ReAct
👥 Complexity & Scale
  • Does this task require multiple different skills?
  • Would different specialists handle different parts?
  • Is there a workflow with parallel or sequential steps?
  • Is this too complex for a single agent?
Points to: Planning

Patterns Combine in Production

In reality, you almost never use just one pattern. A production agent typically combines 2-3 patterns.

Real-World Scenario Pattern Combination
Customer support chatbot RAG Tool Use (search knowledge + create tickets)
Code review assistant ReAct Tool Use (reason about code + run tests)
IT incident response Planning Tool Use RAG (orchestrate investigation)
Data analysis pipeline Planning ReAct (plan analysis steps, reason about results)

The 3 Scenarios

🤖 Scenario A: "The Forgetful HR Bot"

A German manufacturing company (500 employees) wants a chatbot for their HR department. Employees ask questions about vacation policies, sick leave, benefits, and internal regulations.

Key facts:

  • 47 policy documents (PDF)
  • Documents change quarterly
  • No actions needed — just answers

📡 Scenario B: "The Night Shift Monitor"

A logistics company runs 3 warehouses with IoT sensors. They want an AI agent that monitors sensor data overnight, detects anomalies (temperature spikes, unusual movement), and decides whether to:

  • (a) log it
  • (b) alert the night shift manager
  • (c) trigger emergency shutdown

Constraint: Response time must be under 30 seconds.

📋 Scenario C: "The Audit Nightmare"

A financial services company faces EU AI Act compliance audit in 6 months. They have 12 AI agents already deployed across departments. None have governance documentation.

They want a system that:

  • Audits each agent
  • Creates governance docs
  • Identifies compliance gaps
  • Generates remediation plan with priorities
💡

Don't just accept the first answer from your AI Tutor. Ask it to explain why it chose that pattern. Connect it back to what governance constraints would apply!
